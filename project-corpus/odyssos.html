Problem

Build a long‑form idle/incremental game that stays engaging well past the first few hours by combining classic generator training with a map‑based exploration layer, meaningful meta‑progression (artifacts, skills, rebirths), and quality‑of‑life systems (offline progress, autobuy, time‑based rewards) — all with approachable web tech and a content pipeline that’s easy to expand and balance.

Approach

Odyssos is structured around a modular, event‑driven core with a data‑first content model. Nearly all content — trainings, generators, upgrades, worlds/regions/zones, artifacts, skills, and achievements — is defined in JSON. A central mod system composes effects across features in a predictable priority order, while state and UI managers keep the game loop responsive and the interface dynamic.

Architecture

  • Core loop + managers — _Game.js orchestrates _GameManager.js (content + calculations), _StateManager.js (save/load, versioning), and _GameUI.js (DOM skeleton, updates, tooltips).
  • Event bus — _EventManager.js decouples UI, gameplay, and persistence via a lightweight observer pattern.
  • Data‑driven content — JSON under assets/gameData/ powers trainings, upgrades, worlds/regions/zones, artifacts, skills, fighters, tabs, and interface structure.
  • Mod system — composable, priority‑based modifiers apply production, cost, and global effects to specific targets or whole types (e.g., allTrain, forceTrain).
  • State + compression — saves serialize to JSON, then compress with lz-string for localStorage; supports rebirth and reset flows with simple version checks.
  • UI skeleton — layout is generated from interfaceElementData.json; per‑tab managers handle Training, Exploration (Odyssey), Forge, Skills, Essence, Achievements, Settings, and Radiance.
  • Exploration (Odyssey) — worlds → regions → zones form a directed graph with progression bosses, legendary bosses, and rewards; auto‑ conquest respects “beaten once” rules.
  • Tournament — tiered fighters pay out Crystal for powerful forge upgrades; headband rank tracks progress.
  • Offline + rewards — dedicated managers compute idle gains and manage hourly/daily rewards without blocking the main loop.

Stack

  • Vanilla JavaScript (ES modules) — core game logic and UI
  • Parcel — zero‑config bundling for local dev
  • break_eternity — fast big‑number math for idle pacing
  • lz‑string — compressed saves in localStorage
  • Lodash — utilities where helpful
  • HTML/CSS — data‑driven, dynamic DOM generation

Challenges

  • Progression balance — many systems (training, zones, artifacts, skills, tournament) interact; the priority‑based mod tree and JSON content let balance iterate quickly without code churn.
  • Performance + memory — UI updates and large data sets required careful caching; memory leak issues were mitigated with controlled reloads during rebirth/reset flows and targeted refactors.
  • State size + integrity — compressed saves, per‑feature state modules, and simple version checks keep persistence fast and compatible.
  • Dynamic UI complexity — a data‑driven skeleton and small UI managers (tabs, hotkeys, multipliers, notifications) keep the DOM predictable.

Outcomes

Reached a stable, content‑rich alpha (v0.5.x) with a complete Training loop, Odyssey exploration across multiple worlds/regions, a tiered Tournament, artifacts and a skill tree, offline rewards, and autobuy. Major refactors improved clarity and performance; the current structure supports fast content authoring and future rebirth layers (Godhood, Transcendence) outlined in the roadmap.
